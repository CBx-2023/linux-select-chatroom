# 基于 TCP Socket 与 select 的多人聊天室系统设计文档

## 1. 项目背景

本项目用于《Linux 高级编程》课程设计，选题方向为网络类。系统通过 Linux Socket 编程实现一个服务端和一个客户端，支持多个客户端同时连接到同一个聊天室，并完成群聊、私聊、在线用户查询、用户退出和服务端日志记录等功能。

项目使用 C++ 开发，服务端必须运行在 Linux 系统上。代码最终放置在 `code` 目录下，并配套提交课程论文。论文中应包含关键代码截图、系统运行截图、核心流程说明和测试结果。

## 2. 设计目标

系统目标如下：

- 实现一个 TCP 聊天室服务端和一个 TCP 聊天室客户端。
- 服务端使用单进程加 `select` I/O 多路复用模型管理多个客户端连接。
- 客户端登录时设置昵称，服务端维护在线用户列表。
- 支持群聊消息广播。
- 支持用户之间一对一私聊。
- 支持查询当前在线用户。
- 支持正常退出和异常断开处理。
- 服务端记录登录、退出、异常断开和聊天消息等关键事件。

不在本次设计范围内的功能：

- 不实现账号密码认证。
- 不使用数据库。
- 不实现离线消息。
- 不实现历史聊天记录查询。
- 不实现图形界面。
- 不实现管理员权限和踢人功能。

## 3. 总体架构

系统由两个可执行程序组成：

- `server`：聊天室服务端，负责监听端口、接受客户端连接、维护在线用户状态、解析客户端命令、转发群聊和私聊消息、写入运行日志。
- `client`：聊天室客户端，负责连接服务端、发送用户输入、接收服务端消息，并将服务端返回内容展示给用户。

服务端采用单进程事件循环。所有客户端连接都被看作 Linux 文件描述符，服务端使用 `select` 同时监听监听 socket 和已连接客户端 socket。当监听 socket 可读时，说明有新客户端连接；当客户端 socket 可读时，说明该客户端发送了消息或断开了连接。

该架构重点体现以下 Linux 高级编程知识点：

- TCP Socket 创建、绑定、监听和连接。
- 文件描述符管理。
- `select` I/O 多路复用。
- 服务端并发连接处理。
- 基于文本协议的客户端和服务端协作。
- 网络异常和资源清理。

## 4. 通信协议

客户端和服务端之间使用基于文本行的简单协议。每条消息以换行符结束，便于调试、解析和论文说明。

客户端发送命令：

```text
LOGIN <nickname>
MSG <content>
PRIVATE <target_nickname> <content>
LIST
QUIT
```

服务端返回或广播消息：

```text
OK <message>
ERROR <reason>
SYSTEM <message>
CHAT <nickname> <content>
PRIVATE <from_nickname> <content>
USERLIST <nickname1>,<nickname2>,...
```

示例交互：

```text
LOGIN alice
MSG 大家好
PRIVATE bob 你好
LIST
QUIT
```

协议规则：

- 客户端连接后必须先发送 `LOGIN <nickname>`。
- 昵称长度为 1 到 20 个字符，只允许英文字母、数字和下划线，且不能与当前在线用户重复。
- 单条协议消息最大长度为 1024 字节，超过限制时服务端返回 `ERROR message too long` 并丢弃该消息。
- 服务端和客户端都按换行符拆分消息，并为每个连接维护接收缓冲区，用于处理 TCP 半包和粘包。
- 未登录用户只能发送 `LOGIN` 或 `QUIT`。
- `MSG` 表示群聊消息，由服务端广播给其他在线用户。
- `PRIVATE` 表示私聊消息，只发送给指定目标用户。
- `LIST` 表示请求当前在线用户列表。
- `QUIT` 表示客户端主动退出聊天室。

## 5. 服务端模块设计

### 5.1 ChatServer

`ChatServer` 是服务端主控模块，负责：

- 创建监听 socket。
- 设置地址复用选项。
- 绑定 IP 和端口。
- 调用 `listen` 进入监听状态。
- 初始化 `fd_set`。
- 运行 `select` 主循环。
- 接收新客户端连接。
- 将客户端消息交给协议解析和消息路由模块处理。
- 在客户端退出或断开时关闭 socket 并清理状态。

### 5.2 ClientManager

`ClientManager` 负责维护客户端状态，主要数据包括：

- 客户端 socket fd。
- 客户端昵称。
- 客户端登录状态。

它提供以下能力：

- 添加新客户端。
- 删除已断开的客户端。
- 根据 fd 查询客户端。
- 根据昵称查询客户端。
- 检查昵称是否重复。
- 生成在线用户列表。

### 5.3 Protocol

`Protocol` 负责解析和生成协议消息：

- 将客户端文本命令解析成命令类型和参数。
- 检查命令格式是否合法。
- 生成 `OK`、`ERROR`、`SYSTEM`、`CHAT`、`PRIVATE`、`USERLIST` 等响应文本。

### 5.4 MessageRouter

`MessageRouter` 负责消息分发：

- 群聊消息广播给除发送者以外的在线用户。
- 私聊消息发送给指定目标用户。
- 系统消息广播给所有在线用户。
- 在线用户列表只返回给请求者。

### 5.5 Logger

`Logger` 负责将关键事件写入服务端日志文件，例如：

- 服务端启动。
- 用户登录。
- 用户退出。
- 客户端异常断开。
- 群聊消息。
- 私聊消息。
- 重要错误信息。

日志文件保存为 `server.log`，可用于论文中的运行过程说明和测试结果佐证。

## 6. 客户端模块设计

### 6.1 ChatClient

`ChatClient` 是客户端主控模块，负责：

- 创建 TCP socket。
- 连接服务端 IP 和端口。
- 发送登录命令。
- 同时处理用户输入和服务端消息。
- 在用户退出时发送 `QUIT` 并关闭 socket。

客户端使用 `select` 同时监听标准输入和服务端 socket。这样用户可以一边输入消息，一边接收其他用户发来的聊天内容。

### 6.2 InputHandler

`InputHandler` 将用户输入转换为协议命令：

- 普通文本转换为 `MSG <content>`。
- `/w 用户名 内容` 转换为 `PRIVATE <target_nickname> <content>`。
- `/list` 转换为 `LIST`。
- `/quit` 转换为 `QUIT`。

### 6.3 Display

`Display` 负责展示服务端消息：

- `SYSTEM` 显示为系统通知。
- `CHAT` 显示为群聊消息。
- `PRIVATE` 显示为私聊消息。
- `USERLIST` 显示为在线用户列表。
- `ERROR` 显示为错误提示。
- `OK` 显示为普通确认信息。

## 7. 运行流程

服务端运行流程：

1. 启动服务端，读取端口参数或使用默认端口。
2. 创建监听 socket。
3. 绑定端口并开始监听。
4. 进入 `select` 主循环。
5. 如果监听 socket 可读，调用 `accept` 接收新连接。
6. 如果客户端 socket 可读，调用 `recv` 读取消息。
7. 如果读取到 0 字节，说明客户端断开连接，清理客户端状态。
8. 如果读取到命令文本，解析命令并执行对应业务逻辑。
9. 根据业务逻辑向一个或多个客户端发送响应。
10. 持续运行直到服务端被手动终止。

客户端运行流程：

1. 启动客户端，输入服务端 IP、端口和昵称。
2. 创建 socket 并连接服务端。
3. 发送 `LOGIN <nickname>`。
4. 登录成功后进入交互循环。
5. 用户输入普通文本时发送群聊消息。
6. 用户输入 `/w` 命令时发送私聊消息。
7. 用户输入 `/list` 时请求在线用户列表。
8. 用户输入 `/quit` 时发送退出命令并关闭连接。
9. 如果服务端断开连接，客户端提示后退出。

## 8. 错误处理

服务端错误处理策略：

- 昵称为空：返回 `ERROR nickname required`。
- 昵称包含非法字符或长度超过限制：返回 `ERROR invalid nickname`。
- 昵称重复：返回 `ERROR nickname already used`。
- 单条消息超过 1024 字节：返回 `ERROR message too long`。
- 未登录发送聊天命令：返回 `ERROR please login first`。
- 私聊目标不存在：返回 `ERROR user not online`。
- 命令格式错误：返回 `ERROR invalid command`。
- 客户端异常断开：关闭 fd，移除用户，写日志，广播下线通知。
- `select`、`accept`、`recv`、`send` 失败：输出错误信息，必要时清理对应连接。

客户端错误处理策略：

- 连接失败时提示服务端地址或端口错误。
- 登录失败时显示服务端返回原因。
- 服务端断开连接时提示连接已关闭。
- 用户输入空消息时不发送。
- 私聊命令参数不足时在本地提示格式错误。

## 9. 测试方案

功能测试：

- 单客户端登录、发送群聊消息、退出。
- 多客户端同时登录，验证群聊广播。
- 两个客户端之间发送私聊，验证只有目标用户收到消息。
- 使用 `/list` 查询在线用户。
- 使用重复昵称登录，验证服务端拒绝。
- 未登录时发送聊天命令，验证服务端拒绝。
- 输入非法命令，验证服务端返回错误。

异常测试：

- 客户端直接关闭终端，验证服务端能检测异常断开。
- 客户端断开后，其他客户端仍可继续聊天。
- 私聊不存在的用户，验证服务端返回错误。
- 发送超过 1024 字节的消息，验证服务端返回错误且系统不会崩溃。

运行结果检查：

- 观察多个客户端终端中的消息显示是否正确。
- 检查服务端终端输出。
- 检查服务端日志文件中是否记录关键事件。
